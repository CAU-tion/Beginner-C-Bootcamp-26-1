## Challenge1. Format String Bug 실습

### 실습
이번 실습에서는 `printf(argv[1]);`와 같이 사용자 입력을 그대로 출력하는 코드를 실행해보고 
포맷 지정자를 입력했을 때 프로그램이 어떻게 동작하는지 확인하였다.  
이후 `printf("%s", argv[1]);` 형태로 수정하여 안전한 출력 방식과 비교하였다.

---

### 정리
프로그램 실행 시 입력값에 따라 다음과 같은 현상이 발생하였다.

- `%x` → 스택에 저장된 값들이 16진수 형태로 출력됨  
- `%p` → 메모리 주소값 출력  
- `%d` → 정수 형태로 값 출력  
- `%s` → 문자열로 해석되면서 비정상적인 출력 또는 오류 발생

반면, 안전한 코드에서는 어떤 입력을 넣어도 입력 문자열 그대로 출력되었다.

`printf`는 첫 번째 인자를 단순 문자열이 아니라 출력 형식으로 해석한다.  

즉, `%x`, `%p` 같은 문자를 만나면 추가 인자가 전달되었다고 가정하고  
스택 영역에서 값을 꺼내 출력한다.

하지만 `printf(argv[1]);`에서는 실제로 추가 인자가 존재하지 않기 때문에 
스택에 남아있는 임의의 값이 읽혀 출력된다.

---

### 결론
- 사용자 입력을 포맷 문자열로 사용하는 것은 매우 위험하다  
- 이는 프로그램 내부 메모리를 외부에 노출시키는 취약점으로 이어진다

## Challenge2. 반복문 3종 비교

### 실습
1부터 10까지의 합을 계산하는 동일한 기능을  
`for`, `while`, `do-while` 세 가지 방식으로 각각 구현하였다.

---

### 실행 결과
세 방식 모두 동일하게 최종 합 `55`를 출력하였다.

---

### 정리

- `for`
  - 반복에 필요한 요소(초기화, 조건, 증가)가 한 곳에 모여 있음
  - 코드 흐름이 직관적
  - 반복 횟수가 명확할 때 사용

- `while`
  - 조건만을 중심으로 반복
  - 초기화와 증가를 따로 작성해야 함
  - 조건을 중심으로 반복할 때 사용

- `do-while`
  - 먼저 실행 후 조건을 검사
  - 조건과 관계없이 최소 1회 실행됨
  - 무조건 1번은 실행이 필요할 때 사용

## Challenge3. 조건문으로 분기 구현

### 실습
입력된 정수를 기준으로 양수, 음수, 0을 판별하는 프로그램을  
여러 방식으로 구현하였다.

---

### 정리
- `if-else`
  - 범위를 기준으로 조건을 나누는 데 가장 적합  

- `?:` (삼항 연산자)
  - 간단한 조건을 짧게 표현 가능  
  - 중첩 시 가독성이 떨어짐  

- `switch`
  - 직접적인 범위 비교는 불가능  
  - 조건 결과를 값으로 변환하여 처리

## Challenge4. gdb로 반복문 뜯어보기

### 실습
`week03-while.c`를 `gcc -g -O0`으로 컴파일 후 gdb를 사용하여 디버깅을 해보았습니다.

---

### 정리
- `break` → 특정 위치에서 실행 중단
- `run` → 프로그램 실행
- `next` → 다음 줄 실행
- `step` → 함수 내부까지 실행
- `print` → 변수 값 확인
- `info registers` → 레지스터 상태 확인
- `x/` → 메모리 값 확인

---

## 전체 정리

이번 실습을 통해 다음과 같은 내용을 확인할 수 있었다.

- 잘못된 `printf` 사용은 메모리 유출 취약점으로 이어질 수 있음  
- 반복문은 목적에 따라 적절한 구조를 선택해야 함  
- 조건문은 상황에 따라 가독성과 표현력을 고려해 사용해야 함  
- gdb를 활용하면 프로그램 내부 동작을 구체적으로 분석할 수 있음
