## Challenge1. 자료형 크기 확인

### 실습 내용
`sizeof` 연산자를 이용하여 C의 기본 자료형(`char`, `short`, `int`, `long`, `long long`, `float`, `double`)의 크기를 확인하고,  
`signed`와 `unsigned` 자료형의 차이를 비교하였다.

### 핵심 개념
- **자료형(Data Type)** : 변수에 저장할 값의 종류를 정의
- **sizeof** : 자료형 또는 변수의 메모리 크기(byte)를 확인하는 연산자
- **signed** : 음수와 양수를 모두 표현
- **unsigned** : 음수를 표현하지 않는 대신 더 큰 양수 범위를 가짐

### 왜 이런 크기인가
자료형의 크기는 **표현 가능한 값의 범위와 정밀도**에 따라 결정된다.

- 정수형은 더 큰 값을 저장할수록 더 많은 바이트를 사용한다.
- 실수형은 소수점 이하까지 표현해야 하므로 더 많은 비트를 사용한다.
- `signed`와 `unsigned`는 크기는 같지만, 표현 방식이 달라 범위가 달라진다.
- 자료형의 크기는 시스템 환경(32bit / 64bit)에 따라 달라질 수 있다.

## Challenge2. 비트 연산

### 실습 내용
두 정수를 입력받아 AND, OR, XOR, NOT, shift 연산을 수행하고 결과를 출력하였다.

### 핵심 개념
컴퓨터는 모든 데이터를 **2진수(0과 1)** 형태로 저장한다.  
비트 연산은 이 2진수의 각 자리(bit)를 기준으로 수행되는 연산이다.

- AND(&): 두 비트가 모두 1일 때만 1  
- OR(|): 하나라도 1이면 1  
- XOR(^): 서로 다를 때 1  
- NOT(~): 모든 비트를 반전  
- Shift(<<, >>): 비트를 왼쪽/오른쪽으로 이동

## Challenge3. Integer Overflow

### 실습 내용
INT_MAX에 1을 더하여 overflow를 확인하고,  
signed int와 unsigned int의 결과를 비교하였다.

### 핵심 개념
- 자료형은 표현 가능한 값의 범위가 제한되어 있음  
- 이 범위를 초과하면 **overflow(오버플로우)** 발생  

### 실험 결과
- INT_MAX + 1 → 음수처럼 변함  
- UINT_MAX + 1 → 0  

### 원인
컴퓨터는 정수를 **고정된 비트 수(예: 32비트)**로 저장한다.  
따라서 표현 가능한 최대값을 넘어가면 더 이상 저장할 수 없어 값이 깨진다.

- **unsigned int**
  - 범위를 넘으면 **0부터 다시 시작 (순환 구조)**
  - 예: 최대값 + 1 → 0

- **signed int**
  - overflow 시 결과는 **정의되지 않은 동작 (Undefined Behavior)**
  - 대부분의 시스템에서는 음수처럼 보이지만, 항상 그런 것은 아님

즉, overflow는 프로그램의 예측 불가능한 동작을 유발할 수 있다.

## Challenge4. Format String

### 실습 내용
printf의 포맷 지정자(`%d`, `%x`, `%c`, `%s`, `%p` 등)를 사용하여 같은 값을 다양한 방식으로 출력하고,  
잘못된 포맷 지정자를 사용했을 때의 결과를 확인하였다.

### 핵심 개념
`printf`는 포맷 지정자에 따라 **메모리에 저장된 값을 해석하는 방식이 달라진다.**

- `%d` : 정수 (10진수)
- `%x` : 정수 (16진수)
- `%c` : 문자
- `%s` : 문자열
- `%p` : 주소(포인터)

따라서 같은 값이라도 포맷에 따라 완전히 다르게 출력된다.

---

### 잘못된 포맷 사용 시 문제

자료형과 맞지 않는 포맷을 사용하면:

- 잘못된 값 출력
- 메모리를 잘못 읽음
- 프로그램 오류 또는 비정상 동작 발생

예:
- int 값을 `%s`로 출력 → 문자열 주소로 해석 → 이상한 값 출력

---

### 보안 취약점

사용자 입력을 그대로 `printf(user_input);`에 넣으면, 입력값이 문자열이 아니라 포맷 명령으로 해석될 수 있다.  
이 경우 공격자가 `%x`, `%s`, `%n` 등을 이용해 메모리 정보를 읽거나 수정할 수 있는데, 이를 **포맷 스트링 취약점**이라고 한다.
